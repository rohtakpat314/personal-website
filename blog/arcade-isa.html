Building a Playable Arcade Game on an 8×8 LED Matrix: Why Custom ISAs Matter

Embedded Systems Custom ISA ATmega ECE 210 PCB Design

I built a working arcade game on an 8×8 NeoPixel LED matrix for my ECE 210 final project. It plays Snake, Pong, and Flappy Bird — all on a display the size of a postage stamp. But here's the thing: I didn't just code a game and wire it up. I designed a custom instruction set architecture (ISA) to manage it. This forced me to think about real-time constraints, state management, and what it actually means to build embedded systems from scratch.

Most of what I'd done before was firmware, digital systems design, or systems thinking. This project was different — it let me solder, optimize physical space, and debug real electronics. That hands-on constraint is what made it click.

The Problem: 8×8 Is Impossibly Small

An 8×8 LED grid is tiny. 64 pixels. That's it.

The real problem wasn't the LEDs — it was the question: how do you make something fun on a postage-stamp display?

Blinking LEDs is easy. Making a playable game is different. You need:

  • Game logic: Snake dies if it hits a wall. Eating an apple grows the snake by 1. These aren't trivial state transitions on an MCU.
  • Real-time rendering: Your display refresh has to feel smooth, not flickery. Get the timing wrong and the game feels broken.
  • User input handling: Two buttons need to control multiple games with different input schemas. Up/down for Snake. Left/right paddles for Pong. That's a mapping problem.
  • Multi-game switching: Rather than flashing new firmware for each game, I wanted a console-like experience — scroll through games, pick one, play.

This is where a custom ISA came in. Instead of writing spaghetti code with 50 if-statements, I could define instructions for game control, clearing the display, showing winners, and managing state transitions.

The Constraint: Only Two Buttons, One MCU, Limited Memory

Here are the hard limits:

  • ATmega328P microcontroller (32KB flash, 2KB RAM)
  • Two push buttons (digital inputs only)
  • 64 LEDs (addressable, requiring precise SPI-like timing)
  • 7 hours in the lab (single soldering session to get it right)

The ATmega328P is a workhorse, but it's tight. I had to think ruthlessly about memory. Every array, every string, every game instance had a cost.

The two buttons were the real constraint. Pong needs 4 inputs (L/R for each player). I only had 2. So I automated one paddle — it moves autonomously in one direction, and the player controls the other direction. It's a limitation that became elegant.

The Solution: A Custom 8-Bit ISA

I designed an 8-bit instruction set with a clean separation between:

  • Game logic (handled in C++)
  • Display control (ISA instructions)
  • State management (one-hot state machine, like a real console)

ISA Overview

Each instruction was 8 bits. I didn't need many:

  • Clear display
  • Draw pixel
  • Set game state
  • Control flow (unconditional jumps)
  • Start/end program

The clever bit: I used unconditional instructions (like jumps) for game state transitions in a way I'd never done before. Instead of nested if-else logic, I could define clean state paths.

State: MENU
  Wait for input
  If button_A pressed → jump to SNAKE_INIT
  If button_B pressed → jump to PONG_INIT

State: SNAKE_INIT
  Clear display
  Initialize snake at center
  Set state to SNAKE_RUNNING
  ...

It's verbose, but the verbosity is the point. It forces clarity.

Rendering Pipeline

The hard part: updating 64 LEDs in real-time without flicker.

I flowcharted the entire rendering process:

  1. Update game state (snake position, score, etc.)
  2. Clear frame buffer (array of pixel colors)
  3. Render game objects (snake, food, paddles)
  4. Send to NeoPixels (SPI-like protocol, precise timing)
  5. Repeat at ~60 FPS

If your timing is off by a few microseconds, the LEDs glitch. If your refresh rate is too slow, the game feels laggy. I had to be precise.

Input Handling

Two buttons, unlimited possibilities:

  • Snake: Button_A = up, Button_B = down
  • Pong: Button_A = one player's paddle, Button_B = other (with automation)
  • Flappy Bird: Button_A = flap

The buttons could map to anything. This flexibility was surprising — once I soldered them in place, I realized how free the design suddenly felt.

The Build: 7 Hours in the Lab

Hours 0–2: Layout and planning. I spent an embarrassing amount of time thinking about the most space-efficient way to solder components. ATmega328P in the center, NeoPixel matrix on the right, buttons on the left.

Hours 2–5: Soldering. Basic components went down fine. Wires kept coming loose — I'd solder them, they'd detach. After maybe 20 attempts, I learned to use a copper wire to desolder and clean, then reattempt. Tedious, but necessary. The real pain: connecting 8×8 LEDs to the MCU. Three wires (5V, GND, DIN) carrying precise digital signals. One cold joint = the whole matrix dies.

Hours 5–7: Integration and debugging. Code was already written (I'd tested it in simulation), so this was mostly wiring verification.

  • First bug: Pong required 4 user inputs but I only had 2 buttons. Solution: automate one paddle. Elegant under constraint.
  • Second bug: I soldered the DIN input on the NeoPixel at the very end. Suddenly worried I'd shorted the entire matrix. Held my breath. Powered it on. It worked.
  • Third bug: Initial versions had flickering. Turned out my refresh rate was inconsistent. Tightened up the timing logic.

By hour 7, the entire thing worked end-to-end.

The Result

The game runs. You can navigate a menu to pick Snake, Pong, or Flappy Bird, play multiple rounds, see a winner screen for Pong, and return to the menu.

  • Refresh rate: ~60 FPS (smooth, no visible flicker)
  • Button latency: <50ms (button press to LED response)
  • Total power draw: ~500mA (NeoPixels are power-hungry)

What I'd do differently:

  1. Bigger display — 8×8 is great for learning, but 16×16 would be transformative
  2. External controller — a proper joystick would let the game breathe
  3. Wireless input — Bluetooth or IR instead of hardwired buttons
  4. Better intro UX — loading screens, animations, clear instructions

What I Learned

Custom ISAs Are Powerful

ISAs feel like overkill for a small project. They're not. Defining an instruction set forced me to think clearly about the abstraction between game logic and display control. Rather than scattered firmware commands, I had a clean, interpretable sequence of instructions. This scales.

The ATmega328P Has Real Limits

32KB flash and 2KB RAM are tight for anything with game logic, rendering, and multiple states. I had to minimize string literals, reuse buffers, and think carefully about data structures. For a real project, I'd step up to an STM32 or similar.

Embedded Systems Are About Constraint

You have limited power, limited memory, limited pins. The art is making something good within those constraints. The two-button Pong limitation forced me to automate the opponent's paddle — which ended up being more elegant than the original design. Constraints breed creativity.

Soldering Is a Skill

I knew this intellectually. Doing it for 7 hours taught me it viscerally. Cold joints are invisible and infuriating. Desoldering and reattempting 20 times sucks. But by the end, I was confident. That matters.

What's Next?

Could I implement this on a bigger matrix? Add wireless input via Bluetooth? Design a proper game console around this architecture? Port this to a more powerful MCU with complex games?

The arcade game itself is a demo. The real learning was the design methodology: custom ISA, state machines, real-time constraints, and physical optimization under pressure. That methodology applies to anything.
